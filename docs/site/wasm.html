Wasm

Run Vugra in a browser

The current browser path compiles a `.vue` component into Go wasm and renders Vugra commands into a `` through the wasm canvas host.

Build a Bundle

Use the high-level `wasm` command from the repository root:

go run ./cmd/vugra wasm examples/counter/Counter.vue /tmp/vugra-counter-wasm

The same command accepts a project directory or config file:

go run ./cmd/vugra wasm examples/finder /tmp/vugra-finder-wasm
go run ./cmd/vugra wasm examples/finder/vugra.config.json /tmp/vugra-finder-wasm

Project inputs resolve `entry` from `vugra.config.json` and use `app.title`, `app.width`, `app.height`, and `runtime.layout` for the generated host page, canvas runtime constraints, and layout engine selection.

The output directory contains three files:

FilePurpose
index.htmlStatic host page with a canvas and Go wasm loader.
app.wasmCompiled Vugra component app for `GOOS=js GOARCH=wasm`.
wasm_exec.jsGo runtime JavaScript copied from the local Go toolchain.

Serve Locally

Build and serve a temporary preview bundle in one command:

go run ./cmd/vugra wasm-run examples/counter/Counter.vue

Use Vugra's static bundle server to preview an existing output directory:

go run ./cmd/vugra wasm-serve /tmp/vugra-counter-wasm

Any static file server also works for the same output directory:

python3 -m http.server 8000 --directory /tmp/vugra-counter-wasm

Open `http://localhost:8000`. The server is needed because the generated host page loads `app.wasm` with `fetch()`, and browsers restrict wasm loading from `file://` pages.

Smoke Test

The repository includes a Node-based smoke runner that loads `app.wasm` with Go's `wasm_exec.js`, provides a minimal DOM/canvas host, and verifies startup, status updates, canvas drawing, event listener installation, and device-pixel-ratio scaling.

node tools/wasm-smoke/run.mjs /tmp/vugra-counter-wasm

For a real browser check, the Chrome/CDP runner serves the bundle over local HTTP, waits for the wasm app to report `Running`, verifies non-background canvas pixels, reads the hidden accessibility mirror, and can write a PNG screenshot:

node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --screenshot /tmp/vugra-counter-browser.png

The same browser runner can dispatch real Chrome mouse input and hidden accessibility mirror actions:

node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --click 30,60 --expect-text 1 --expect-a11y button,+
node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --a11y-click button,+ --expect-text 1 --expect-a11y button,+

Accessibility focus is separate from activation: focusing a mirrored node updates Vugra focus state without firing the button, and later key activation still routes through the focused runtime element:

node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --a11y-focus button,+ --expect-text 0 --expect-a11y-focused button,+
node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --a11y-focus button,+ --key Enter --expect-text 1 --expect-a11y-focused button,+

Keyboard focus and activation are checked in Chrome by tabbing to a canvas-rendered button and dispatching Enter or Space through the wasm key bridge:

node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --key Tab --key Enter --expect-text 1 --expect-a11y-focused button,+
node tools/wasm-browser-check/run.mjs /tmp/vugra-counter-wasm --key Tab --key ' ' --expect-text 1 --expect-a11y-focused button,+

Event Bridge

The generated wasm main installs event bridges from browser input into Vugra runtime methods:

Browser InputRuntime Dispatch
mouse down / clickDispatchPointerEvent
mouse moveDispatchHover
dragDispatchDrag
double clickDispatchDoubleClick
context menuDispatchContextMenu
wheelDispatchScroll
keyboard, caret/selection editing, IME composition, and pasteDispatchKey and DispatchTextInput

The canvas host keeps Vugra coordinates in logical CSS pixels. At runtime it sizes the canvas backing store with `devicePixelRatio` and applies a canvas transform, so high-DPI displays get sharper output while pointer and scroll coordinates still match layout coordinates. Browser wasm layout uses a canvas-backed measurer, so text wrapping and text drawing share the browser's `measureText()` metrics path.

The wasm host also mirrors Vugra's renderer-neutral accessibility tree into a hidden DOM container. The mirror exposes roles, names, checked state, disabled state, focused state, and layout metadata for the current frame while the visible UI continues to render through canvas. Focus, click, key, and text actions from the mirror are translated back into Vugra runtime dispatch.

Native and wasm hosts share the same runtime key names for focus traversal and text editing, including Tab, Shift+Tab, ArrowLeft, ArrowRight, Home, End, Backspace, Delete, Enter, Space, Escape, and Mod+A.

Verified Examples

The current bundle workflow has been verified by command-line build and smoke execution for:

go run ./cmd/vugra wasm examples/counter/Counter.vue /tmp/vugra-audit-counter
go run ./cmd/vugra wasm examples/finder /tmp/vugra-audit-finder

Both commands generate `index.html`, `app.wasm`, and `wasm_exec.js`.

Current Limits

Implemented Canvas 2D rendering, canvas-backed wasm text measurement, generated wasm main, host page generation, pointer/scroll/keyboard/text/caret/selection/IME composition/paste bridges, hidden DOM accessibility mirror with focus/click/key/text action bridges, inline and referenced SVG subset rendering, reusable bundle command, and Chrome/CDP browser QA with screenshot output.

Still future work WebGPU wasm rendering, stricter native/browser font parity guarantees, richer platform accessibility actions beyond activation and text input, and a production widget library.

Wasm support is browser-runnable through canvas today; WebGPU/Vello wasm remains a separate backend milestone.
